CS 370 Portfolio Reflection
Project Two: Pirate Intelligent Agent
For this project, I built a pirate intelligent agent that learns to navigate a maze and find treasure using deep Q-learning. The starter code I was given included the maze environment, the experience replay class, and the overall training loop structure. What I completed was the Q-training algorithm itself, the core logic that lets the agent learn from experience. That included implementing the neural network model, setting up the epsilon-greedy exploration strategy, connecting the replay buffer to the training process, and tuning the hyperparameters to get the agent to consistently solve the maze. The final network used PReLU activation, trained over 1000 epochs with a batch size of 32, and used epsilon decay from 1.0 down to a minimum of 0.05 to balance exploration early on with exploitation as training progressed.
Computer scientists solve problems by building systems that can process information and make decisions at a scale humans cannot match alone. That matters because those systems end up embedded in things people rely on, navigation, medical diagnostics, financial tools, and increasingly autonomous agents like the one in this project. What I took from this course is that approaching a problem as a computer scientist means being precise about what you want the system to learn, not just what you want it to do. In the pirate agent project, the difference between a reward function that worked and one that did not was not obvious at the start. Getting the agent to actually learn required thinking carefully about what feedback it was receiving at each step and whether that feedback was actually pointing it toward the right behavior.
The ethical side of this work is something I thought about more after working through the agent design. When you build a system that makes decisions autonomously, the end user is trusting that the system will behave reliably and that its failures will not cause harm. For the pirate maze the stakes are low, but the same architecture scales to real applications where a bad decision has real consequences. My responsibility to the end user is to be honest about what the system can and cannot do, and to test it thoroughly enough that I am not shipping something that fails in predictable ways I just did not bother to check. My responsibility to the organization is to document the work clearly enough that someone else can understand, maintain, and improve it, which is part of why this portfolio exists.
